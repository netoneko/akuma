Engineering discipline: meow on a diet

  • Stream, never buffer — tokens processed as they arrive; 16 KB response cap, 1 KB chunks.
  • Sliding context window — context turned out to be the #1 heap driver; the heap peaked when accumulated context peaked, not on any tool call.
  • File-backed tool output — results spill to disk, not the heap.
  • Run as CGI — short-lived process per request; pages reclaimed on exit.

At 4 MB every allocation is visible. The kernel log became a free profiler: idle ~130 allocs/s, a tool call spikes to 1000–3000/s.

What moved the needle: tooling, measured

Same task, same local model (qwen3), three tooling configs:

Metricno memory+ memory+ compact tools
Total tool calls924627
Avg response time18.8 s14.0 s6.4 s
Peak prompt~71K~62K22K

Biggest win was correctness: offloading oversized tool results let a run reach 85K tokens — ~22K past where the model used to emit malformed calls and hang.

Honest caveats: part of the first jump was run-to-run variance; a headline "46K→12K" drop was a tokenizer measurement artifact; paging memory back in can re-inflate context. You only know which knob worked because you measured it three times.

Client tooling pays off twice

LeverOn a local modelOn a metered cloud model
Memory offloadmakes it finish at allfewer input tokens = $ saved
Compact tool schemas−15% prompt every turn−15% on every billed request
Kill stale re-injectionfits the small windowlower latency + cost
Structured write toolavoids JSON-truncation hangsfewer wasted retry turns

Local: tokens are ~free, model is brittle → tooling buys reliability.
Cloud: model is strong, tokens are money → same tooling buys cost & latency.

Caveat: completion-nudge features add turns — only worth it where turns are free (local). Ship context-reduction everywhere; nudges on local.

Making the model fit too: llama.cpp

A 532 MB model serving in a 256 MB VM (~2× model:RAM) — and it answers correctly.

OS side — clean file-page eviction. Read-only model pages page out under pressure and re-fault from disk. Lets an mmap exceed physical RAM.

llama.cpp side — shrink the non-evictable (anon) footprint:

  • --no-repack → drops ~495 MB
  • --kv-cache-file → KV ~14 MB vs ~519 MB
  • -fit off -b 64 -ub 64 → ~489 MB

~15 s/token (disk-bound), zero kernel crashes. Sweet spot: model mostly fits → eviction only trims, no thrash.

Efficient Engineering: 4 MB is enough

BudgetWhat you get
4 MBAn agent that acts. Model is remote.
~64 MBComfortable agent VM + runc containers — one sandbox per task.
~256 MBRun the model locally via llama.cpp. The box thinks for itself — a self-hosted dev box with some assistance (slow, but real).
GBsFull-speed local inference.

Same OS, from "thin remote agent" to "self-hosted dev box with a built-in assistant" — just by moving the model across the network boundary, or not.

How does that compare to Linux?

What it takes to run a working agent, not a bare kernel.

StackPractical RAM
Bare Linux kernel (tinyconfig, no userspace)few MB — does nothing useful
Alpine Linux (minimal, official guidance)~128 MB
Linux + containerd/runc + containerized agent~64–128 MB
Python agent stack on Linuxhundreds of MB
Akuma: full clone + compile + run agent4 MB

Honest caveat: a Linux kernel boots tiny. The gap is userspace — add networking, a shell, a compiler, a container runtime and Linux lands 1–2 orders of magnitude higher. Akuma's 4 MB already includes all of that.

What's real vs. what's hard

  • A 25-min run stayed stable, but the model burned ~19 min probing for tools that don't exist (git, curl, apt-get, chmod). Fix: describe the sandbox in the system prompt.
  • A JSON-escaping bug corrupted a source file; a missing signal-aware exit code hid a compiler crash.
  • These are agent-plumbing and prompt bugs, not memory bugs. The OS held the whole time.

At the edge, the agent's biggest enemy is not knowing what's in its sandbox — not RAM.
